# ISS-096 — Gate G1 (Analysis & Plan)

## Ticket interpretation
In `templates/sprint.html`, render each status count card as a clickable anchor
linking to `/tickets?sprint=<sprint>&status=<status>` when the count is > 0.
Zero counts stay as plain text (no link to an empty list). No backend changes
required — the `/tickets` route already supports `?sprint=&status=` filtering.

## Implementation plan
1. In `templates/sprint.html`, modify each `.count` div in the status cards to
   conditionally render an `<a>` tag:
   ```jinja2
   {% if status_counts["open"] > 0 %}
   <a href="/tickets?sprint={{ selected_sprint }}&status=open" ...>{{ status_counts["open"] }}</a>
   {% else %}
   {{ status_counts["open"] }}
   {% endif %}
   ```
   Apply same pattern for `in-development`, `in-testing`, `resolved`.

2. Write 2 new tests in `tools/tests/test_dashboard_sprint.py`:
   - `test_sprint_non_zero_status_count_is_a_link` — seeds sprint with 1 open
     ticket; asserts `href="/tickets?sprint=...&status=open"` present in response.
   - `test_sprint_zero_status_count_is_not_a_link` — seeds sprint with 0 resolved
     (all open); asserts the resolved count section has no `href` containing `status=resolved`.

## Files to be touched

## Risks and mitigations
- **Risk**: Jinja2 template URL encoding for sprint names with hyphens.
  - **Mitigation**: `tools-N` sprint names are URL-safe; no encoding needed.
- **Risk**: Breaking existing sprint tests by changing card HTML.
  - **Mitigation**: Existing tests check text content (`"open"`, `"resolved"`,
    count values) and progress bar — not link presence/absence. Adding `<a>` tags
    around counts does not break those assertions.

## Acceptance criteria mapping
- AC1: Non-zero counts render as `<a href="/tickets?sprint=...&status=...">` anchors
- AC2: Zero counts render as plain text (no `<a>` tag)
- AC3: Linking from Sprint tab to Tickets tab with correct filters (route already works)
- AC4: All 8 existing sprint tests + all existing ticket dashboard tests pass
