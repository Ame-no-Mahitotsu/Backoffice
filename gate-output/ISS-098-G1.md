# ISS-098 — Gate G1 (Analysis & Plan)

## Ticket interpretation
Make the header+nav bar sticky so it remains visible when the user scrolls any
dashboard page. CSS-only fix: `position:sticky; top:0` on `header` and `nav`
elements (or a wrapping `div`), with correct `z-index` and `padding-top` on
`<main>` to prevent content being obscured on load.

## Implementation plan
1. In `templates/base.html`:
   - Wrap `<header>` and `<nav>` in a `<div class="sticky-top">` with CSS:
     ```css
     .sticky-top { position: sticky; top: 0; z-index: 100; }
     ```
   - Add `padding-top: 0` override — `<main>` already has `padding: 2rem`,
     which is sufficient. No additional padding needed since sticky does not
     remove elements from flow.
   - Alternative (simpler): apply `position:sticky; top:0; z-index:100`
     directly on `header` and `nav` — but only `header` needs to stick
     (header + nav form a visual unit). Wrapping both is cleaner.
2. Write TDD test in `tools/tests/test_dashboard.py`:
   - `test_nav_bar_has_sticky_positioning` — asserts `position:sticky` or
     `position:fixed` appears in the rendered HTML response for `GET /`.
   - This fails before the CSS change (current base.html has no sticky CSS).

## Risks and mitigations
- **Risk**: Sticky nav overlaps page headings on initial load.
  - **Mitigation**: `position:sticky` keeps element in normal flow — no overlap
    unless `<main>` has zero top margin. Current `<main>` has `padding: 2rem`
    which prevents this. Verify AC3 with test.
- **Risk**: Breaking existing nav/layout tests.
  - **Mitigation**: No structural HTML changes — only CSS class additions.
    All existing tests check text content and `href` attributes, not CSS.

## Acceptance criteria mapping
- AC1: Sticky positioning CSS present in rendered HTML
- AC2: CSS-only (`position:sticky`/`position:fixed`) — no `<script>` tags added
- AC3: Nav does not overlap content — `z-index` and flow preserved
- AC4: All existing dashboard tests pass (only CSS added; no HTML structure changed)
